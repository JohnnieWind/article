# AI 写界面，终于需要一份设计说明书了

> 平台：微信公众号 | 字数：约2400字 | 调性：理性深度、技术趋势解读

---

AI coding agent 越来越会写代码了。

你给它一张截图、一个需求、几句产品描述，它很快就能搭出页面：按钮、卡片、列表、响应式布局，看起来都像那么回事。

但用过一段时间的人，大多会遇到同一个问题：第一版还不错，第二版开始走样，第三版之后，整个产品的气质就开始松掉。

今天是深色科技风，明天变成卡片 SaaS 风；这个按钮 8px 圆角，那个面板 24px 圆角；某个页面用了很克制的中性色，另一个页面又忽然出现一片紫蓝渐变。

不是 AI 不努力，而是它缺一份稳定的“设计记忆”。

Google Labs Code 最近开源的 [DESIGN.md](https://github.com/google-labs-code/design.md)，解决的正是这个问题。

它看起来只是一个 Markdown 文件规范，真正有意思的地方却在于：它试图把设计系统变成 coding agent 能长期读取、理解、复用的上下文。

换句话说，过去我们给 AI 写需求文档；现在，我们可能也要给 AI 写一份品牌手册。

## DESIGN.md 到底是什么

DESIGN.md 是一种用来描述视觉身份的文本格式。

它的结构很简单：文件顶部是一段 YAML front matter，用来放机器可读的设计 token；下面是 Markdown 正文，用来写人能读懂的设计理由、品牌气质和使用边界。

前者告诉 agent：“主色是 `#1A1C1E`，正文是 16px，按钮圆角是 8px。”

后者告诉 agent：“为什么要用这个颜色？它应该出现在哪些地方？这个界面应该像一本冷静的建筑杂志，还是像一个轻快的宠物服务应用？”

这两个信息缺一不可。

只有 token，AI 只能照表取值，很容易做出“数值正确但气质不对”的界面。

只有形容词，AI 又会滑向平均值。你说“现代、干净、高级、可信”，它大概率生成一套大家都见过的中性 SaaS 页面。

DESIGN.md 的核心判断是：设计不是一串孤立参数，而是一组有意图的取舍。token 给出锚点，文字解释取舍。

这也是它比普通设计 token 文件更值得关注的地方。

## AI 真正缺的不是颜色表，而是上下文

在传统团队里，设计系统通常存在于 Figma、组件库、设计规范文档和设计师的经验里。

人类协作时，这套系统可以靠会议、评审、口头提醒和长期默契维持。

但 agent 不是团队里的老同事。

它每次接任务，看到的只是当前上下文。上下文里如果没有明确的设计约束，它就会根据训练中的“常见好看界面”来补全。

于是问题来了：AI 会写出一个“合理”的按钮，但它不知道这个产品的按钮为什么不能太圆；它会做一个“漂亮”的 hero，但它不知道这个品牌根本不该有英雄区；它会加一点阴影、渐变和玻璃拟态，因为这些在通用审美里很安全，但对你的产品可能完全不对。

DESIGN.md 的价值，就是把这些隐性判断显性化。

它把“这个产品应该长什么样”从聊天记录里拿出来，放进仓库，变成一个可以被读取、版本管理和验证的文件。

这对 AI 开发特别关键。

因为 AI 写代码时，最怕的不是不会实现，而是不知道边界在哪里。设计边界越模糊，它越容易用通用模板填空。

一份好的 DESIGN.md，本质上是在告诉 agent：哪些选择是这个产品的性格，哪些看起来好看但不能碰。

## 最值得借鉴的是“具体 reference”

DESIGN.md 的哲学说明里有一个很重要的观点：具体参考比形容词更有用。

“现代、简洁、高级”太宽泛了。

但如果你说“像 1970 年代老牌大学研究生课程的纸质讲义”，AI 马上会得到一整套隐含约束：克制、密集、无装饰、偏印刷感、强调阅读，不需要浮夸动效，也不需要营销式大标题。

这就是具体 reference 的力量。

它不是让 AI 复制某个作品，而是给它一个审美坐标。

我们今天写 prompt 时，经常堆很多抽象词：

要高级、要科技感、要年轻化、要有呼吸感、要国际化。

这些词单独看都没错，放在一起却很容易互相抵消。最后 AI 只能往中间值靠，生成一种“看起来没毛病，但没有性格”的东西。

更好的写法是给它一个具体世界：

这是一份给投研团队看的内部情报板，而不是 ToC 数据大屏。

这是一个面向独立书店店主的库存工具，而不是电商后台。

这是一个给研究工程师看的实验日志，不是创业公司官网。

这些描述一旦成立，很多设计选择会自然浮现：字体大小、信息密度、留白程度、颜色克制程度、是否需要插图、按钮应该张扬还是安静。

DESIGN.md 鼓励的正是这种写法：不要只告诉 AI 数值，要告诉它世界观。

## 它让设计系统进入代码仓库

DESIGN.md 还有一个很工程化的特点：它不是单纯的说明文档，而是可以被工具处理。

官方 npm 包 `@google/design.md` 已经提供 CLI。根据 npm 信息，截至 2026 年 6 月 24 日，最新版本是 `0.3.0`。

你可以用它 lint 一个 DESIGN.md 文件，检查结构、token 引用、对比度、缺失字段等问题。

也可以 diff 两个版本，观察 token 层面的变化，判断一次修改是不是引入了设计回退。

还可以导出为 Tailwind v3 配置、Tailwind v4 的 CSS theme，或者 DTCG 设计 token 格式。

这件事很重要。

因为一旦设计规范进入代码仓库，它就可以像代码一样被 review、被比较、被回滚、被自动检查。

过去，设计漂移往往发生在“不知不觉”里。

某个开发为了赶进度加了一个临时颜色，某个页面为了视觉效果用了另一套阴影，某次 AI 修改顺手改了按钮样式。单看每次改动都不大，半年后整个产品就变成了很多风格的拼贴。

如果 DESIGN.md 成为仓库里的标准文件，agent 在改 UI 前先读它，人类在评审 PR 时也能对照它，设计一致性就不再完全依赖记忆和口头约定。

这很像前端工程里曾经发生过的事：格式化、类型检查、lint、测试，把很多“靠人注意”的事情变成了“靠系统提醒”。

DESIGN.md 想做的，是把一部分设计一致性也纳入这条链路。

## 它不会替代 Figma，但会改变 AI 协作方式

需要说清楚的是，DESIGN.md 不是 Figma 的替代品。

Figma 仍然适合做探索、画流程、定义组件、表达复杂交互。视觉设计不是只靠一个 Markdown 文件就能完成的。

DESIGN.md 更像一个桥。

它连接的是设计系统和 coding agent。

对设计师来说，它是一份可以写进仓库的设计意图说明：不只交付画面，也交付判断标准。

对工程师来说，它是一份更稳定的前端上下文：当你让 AI 改页面时，不必每次重新解释品牌风格。

对 agent 来说，它是一套可持续读取的边界：知道什么该保持，什么可以变化。

这意味着未来的 AI 前端协作，很可能不再只是：

“根据这个需求实现页面。”

而是变成：

“先阅读 `DESIGN.md`，遵守现有视觉系统，再实现这个页面。”

这句话看似很小，实际影响很大。

它把 AI 从一次性的页面生成器，往“理解项目长期风格的协作者”方向推了一步。

## 现在使用它，应该注意什么

DESIGN.md 目前仍是 alpha 状态，规范和 CLI 都在持续演进。

所以它更适合被看作一个正在成形的方向，而不是一个完全成熟的行业标准。

如果你想在自己的项目里试试，可以从一个很小的版本开始。

第一，写一个具体的设计 reference。

不要只写“高级简洁”。写清楚这个产品像什么、服务谁、应该让人产生什么感受。

第二，写少量关键 token。

先定义颜色、字体、字号、圆角、间距和常见组件，不要一开始就试图覆盖所有情况。过度复杂的规范会让人和 AI 都不愿意读。

第三，写清楚 do 和 don't。

比如：不要使用渐变；不要使用超过两种字体权重；主色只用于页面最重要动作；卡片只能用边框表达层级，不使用厚重阴影。

第四，把它接进 AI 工作流。

每次让 agent 改 UI 前，明确要求它先读取 DESIGN.md。修改设计系统本身时，也让它说明为什么要改、影响哪些组件。

第五，用 lint 和 diff 做基本守门。

这一步不保证设计一定好，但可以减少引用错误、对比度问题和 token 漂移。

## 真正的变化：设计也变成了上下文工程

过去我们讨论 AI 编程，更多关注代码能力：能不能写组件，能不能修 bug，能不能跑测试。

但在真实产品里，代码只是最终结果。更难的是让 AI 理解约束、延续意图、保持一致。

DESIGN.md 提醒我们：未来的产品工程，可能会越来越像“上下文工程”。

需求需要结构化，架构需要结构化，设计也需要结构化。

不是为了把创造力变成表格，而是为了让创造力能被复用、被传递、被协作。

AI 不怕你给它规则。

它怕的是你没有告诉它，哪些规则真的重要。

DESIGN.md 的出现，说明一个趋势正在变清楚：当 AI 开始参与界面生产，设计系统不能只活在人的脑子里，也不能只活在设计工具里。

它需要进入代码仓库，进入 prompt，进入 agent 的工作流。

以后，一个成熟的 AI 项目里，可能会有 `README.md` 告诉人怎么启动项目，有 `AGENTS.md` 告诉 agent 怎么工作，也会有 `DESIGN.md` 告诉 agent：这个产品，应该以什么样的气质出现。

这不是一个小文件的问题。

这是 AI 时代的设计交接方式，正在重新被发明。
